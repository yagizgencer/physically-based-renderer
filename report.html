**Final Project Report** Student name: Yagiz Gençer Final render ============
Motivation ========== I chose to explore the theme of "reflections" by creating a surreal scene that blends the ordinary with the fantastical. In this scene, two Statues of Liberty, one gold and one silver, face each other and float on the water inside a mug with a wooden boat between them. Additionally, a military aircraft performs aerobatic maneuvers, leaving smoky trails behind it. While the statues seem to almost touch the clouds, it becomes evident that everything is inside a room, with the mug placed on a wooden table. In this way, I aimed to create an intriguing, dreamlike scene by merging real and unreal elements, indoor and outdoor settings, and contrasting scales. In line with the theme, the statues and boat have their reflections on the water in the mug. Furthermore, the clouds, mug, table, statues, and aircraft have all their reflections in the mirror, which is on the table. The scene features environmental lighting, and the surrounding environment—beach and ocean—is visible only through its reflection in the mirror. Finally, the two Statues of Liberty of different material facing each other appear side by side in their mirror reflections, holding their torches up to the sky in unity, enhancing the artistic quality of the scene. Feature 1: Textures ========= For texture, I implemented a "Texture" class in "texture.cpp" and "texture.h" files. The class holds a texture image as an array of Color3f arrays as well as the filename and the dimensions of the image. It also has a "getTextureColor()" method that returns the color of the image given a UV coordinate by applying bilinear interpolation. Furthermore, I implemented a class called "ImgTexture" that inherits from "Texture" class and fills the texture image object by reading from the filename and also applying Gamma correction as suggested in PBRT Edition 3 section 10.4. In my implementation, textures are properties of a scene specified by their filenames as strings. They are stored in a "Texture" vector called "m_textures" and can be obtained using "scene->getTextures()" command in the integrator. In the integrator, whenever a diffuse object intersection is found with its filename not an empty string, we find which texture it is and get the corresponding texture color using the UV coordinate of the intersection and store it in "texture_albedo" variable, which is set to Color3f(-1.0f) by default. Whenever we use "eval()" or "sample()" methods of a BSDF, we call these methods now together with texture_albedo, not just the query record. These new implementations of "eval()" and "sample()" methods call their old versions if they are not implemented for a specific BSDF. If they are implemented (for diffuse BSDFs), they use texture_albedo parameter as their albedo value in their calculations if texture_albedo is not Color3f(-1.0f) (if there is a texture). Feature 3: Image Based Lighting ========= For this feature, I first implemented hierarchical sample warping introduced in Assignment #3's "Hacker Points" section. This is done by first implementing a child class of "Emitter" called "EnvLight". This class first reads the environmental image file in its constructor and then creates and stores a Mipmap of it following the approach in PBRT Edition 3 section 10.4. Note that here I assume the input image is a square image to allow for Mipmap creation, and therefore I resize the lighting images to a square size as a preprocessing step. I store the Mipmap as an array of array of arrays as it is a structure holding multiple images (each being an array of arrays). It also has methods for reading the radiance as a lookup from the highest resolution level of the Mipmap structure given a UV point or $ w_i $ incident direction. This class also has a "sample()" method that given a 2D uniform sample, warps it hierarchically according to the mipmap, turns this warped 2D point to $ w_o $ outgoing sampled vector, reads the radiance and also calculates the PDF value for this sample. In the end, it returns (radiance, wo, pdf) tuple to be used by the integrator. To realize this function, I implemented "squareToHierarchicalWarped()" and "squareToHierarchicalPdf()" functions in "warp.cpp" file, where former warps a 2D point according to the Mipmap and the latter finds the probability of this warping. In the integrator, I have an "envLight" emitter pointer initialized as a null pointer and is filled if there is an environmental lighting image specified in the scene xml file. Then, following the PBRT Edition 3 approach in section 12.6, this environmental light source is imagined to be a sphere of two times the radius of the entire scene that surrounds the scene. My volume path tracer integrator uses "path_ems" approach and the necessary radiance, sampled direction and pdf values are obtained by implemented "sample()" method explained earlier. ## Hierarchical Sample Warping Verification I first provide verification for hierarchical sample warping method using the statistical test I implemented in "warptest.cpp" file. Feature 4: Homogeneous Participating Media ========= To implement this feature, I needed Henyey-Greenstein phase function to model the scattering of the ray inside the homogeneous medium. For this, I implemented "sampleHenyeyGreenstein()" and "pdfHenyeyGreenstein()" functions in "warp.cpp" file. The former returns a outgoing direction $ w_o $ given an incident direction $ w_i $ for a specific value of "g" parameter and the latter returns the probability of sampling this particular outgoing direction $ w_o $ given that $ w_i $ incident direction. For the homogeneous medium itself, I implemented a child class of "Medium" class called "HomogenousMedium" in "homogenous_medium.cpp" file. This class reads albedo, g, and $ \sigma_t $ values for the medium specified in the scene xml file and also calculates $ \sigma_s $ and $ \sigma_a $ values from them. It also has a "sample()" method that selects a channel of 3D $ \sigma_t $, samples a traveled distance within the homogeneous medium and decides if the ray still stays inside the medium or not, and returns Tr (attenuation coefficient) times $ \sigma_s $ divided by PDF or just Tr divided by PDF depending on if the ray is still inside the medium or not respectively. The implementation of this function follows exactly the implementation provided in PBRT Edition 3 section 15.2. I also implemented a function called "rayIntersectTr()" in "scene.h" file. This is a generalization of "rayIntersect()" function that was already implemented in Nori. This function is used during the emitter sampling phase in the integrator and instead of only returning a True or False as previously done with shadow rays, it is designed to also calculate the beam transmittance (or attenuation coefficient) up to the sampled light point if the ray goest through homogeneous media along its way. Basically, it initializes Tr with Color3f(1.0f) and begins a while loop. If the ray enters a homogeneous medium, this does not mean a complete occlusion now but Tr value should be decreased by a value determined by how long the ray has traveled in the medium and what are the attenuation parameters of the medium are. If the ray hits a standard object, Tr is returned as Color3f(0.0f), meaning a full occlusion. If not, a Tr value is returned, which is a constant that should multiply the radiance value provided by the light source as an attenuation factor. The implementation of this function very closely follows "IntersectTr()" function provided in PBRT Edition 3 section 11.3. I also implemented the medium property as an attribute of the current ray by adding a "Medium" pointer attribute to "ray.h" file, which is initialized by a null pointer. This medium pointer is a null pointer if the ray is not fully contained in a medium between its last two intersection points. Otherwise, it keeps the parameters of the medium it is contained in. Lastly, it is important to note that in order to support a shape for the medium, I implemented medium as an attribute of the "Mesh" class. If no medium is specified for a mesh in the xml file, this medium pointer is again a null pointer. Otherwise, the medium of a mesh can be obtained by calling the "getMedium()" function of the mesh. As the last step, I implemented a volumetric path tracer integrator in "path_volume.cpp" file. I keep the "path_ems" (emitter sampling) approach for the integrator but add the medium-awareness capability to it using the methods provided before. For this integrator, I exactly follow the "VolPathIntegrator" provided in PBRT Edition 3 section 15.3. However, the algorithm to be followed can be outlined as follows: 1. We have a bool variable called "insideMedium", and we initialize it by "false" at the beginning of each iteration. This variable, given that the ray is fully contained in the medium (ray.medium is not a null pointer), tells whether we have sampled a point inside the medium so that we should have a scattering inside the medium using the phase function or we have sampled a point outside the medium so that the ray should pass through the medium (of course with attenuation) without being scattered to a new direction. 2. If the ray is contained in the medium (not to mix it up with insideMedium - this just means that we are sure the ray traveled fully inside the medium between its last two intersection points but we will decide if it will be scattered within the medium or not by sampling a distance), we sample a distance using "sample()" method, sample a "t", and update the "insideMedium" bool variable accordingly. Furthermore, we multiply "beta" with the output of this call. Furthermore, we update our new intersection point according to the "t" sampled by traversing this distance in the ray direction. Note that as before, "Le" emitter component is calculated if the intersection point is on the light source and the last bounce is specular. 3. If "insideMedium" is true, we first sample a point from the light source. Then, we use "rayIntersectTr()" method as a shadow ray, which also gives us the attenuation constant Tr. Furthermore, we get the probability of medium scattering the ray in this sample direction by calling "pdfHenyeyGreenstein()" method. Then, "Ld" direct illumination component is calculated using the geometric term, radiance value, probability, and attenuation constant. Now that we obtained the light contributions, we sample a new scattered direction using "sampleHenyeyGreenstein()" method of the medium and replace the ray's direction with this sampled direction. At the end, we leave the loop without any further updates. 4. Otherwise, (if "insideMedium" is false), we first check if the new intersection is with a medium or not. If yes, we first take our ray's origin to this intersection point. If ray.medium was a null pointer before, we take the medium of the intersected mesh as new ray.medium because now the ray will be contained in this medium from that point to its next intersection. If it was not a null pointer, now the ray will leave its medium, so we set ray.medium to a null pointer. Here, if the BSDF of intersection is diffuse, we need to calculate an Ld direct illumination component. This is again done by sampling a light source point. The only difference now is that we use "rayIntersectTr()" method as a shadow ray which also calculates attenuation coefficient Tr for the cases where there happens to be some homogeneous media between the intersection point and the sampled light point. "Ld" component obtained is of course multiplied by this attenuation. 5. The end of the loop is the same as in "path_ems", where we sample a new direction for the ray and update beta, eta, and specBounce variables. ## Henyey-Greenstein Phase Function Verification I first provide verification for Henyey-Greenstein phase function using the statistical test I implemented in "warptest.cpp" file that uses these two functions explained earlier.
